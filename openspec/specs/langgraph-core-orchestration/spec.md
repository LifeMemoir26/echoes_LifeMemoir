## 目的
来自 `change-to-langgraph-core` 的同步规范：约束 LangGraph 核心编排能力。

## 需求

### 需求：主流程必须由 LangGraph 统一编排
系统 应当 将 knowledge、interview、generate 三条主流程的编排收敛为 LangGraph `StateGraph`，并且应用层只通过统一工作流入口触发执行，不得继续新增散落在服务层的独立编排路径。

#### 场景：统一入口触发流程
- **当** 应用层请求执行任一主流程
- **则** 系统通过对应 `StateGraph` 执行并返回流程结果，不经过旧的手工编排链路

#### 场景：边界情况-未知流程标识
- **当** 调用方传入未定义的流程标识
- **则** 系统返回结构化错误并拒绝执行，且不触发任何节点副作用

### 需求：工作流状态契约必须类型化且可校验
系统 应当 为每个工作流定义显式状态契约（TypedDict 或 Pydantic），至少包含 `workflow_id`、`thread_id`、`status`、`errors`、`metadata` 等基础字段，并为节点输入输出定义可校验的 payload 结构。

#### 场景：节点输出状态增量
- **当** 任一节点执行完成
- **则** 节点只返回与契约一致的状态增量字段，且字段类型校验通过

#### 场景：边界情况-状态字段缺失
- **当** 节点返回缺失必填字段的状态
- **则** 系统标记该节点失败并记录可诊断错误，不合并无效状态

### 需求：并行分支汇合必须使用显式 reducer 规则
系统 应当 为并行分支写入的聚合字段定义 reducer（如列表合并、计数累加），禁止依赖隐式覆盖行为，确保结果可预测。

#### 场景：并行分支成功汇合
- **当** 两个及以上并行分支同时写入同一聚合字段
- **则** 系统按 reducer 规则合并结果且不丢失任一分支有效输出

#### 场景：边界情况-缺失 reducer
- **当** 并行写入字段未定义 reducer
- **则** 系统拒绝编译或在运行时以明确错误终止，避免静默覆盖

### 需求：工作流必须支持 checkpoint 与 thread_id 恢复
系统 应当 在编译图时启用 checkpointer，并要求调用方提供稳定 `thread_id` 以支持中断恢复、失败重放和状态追踪。

#### 场景：中断后恢复执行
- **当** 流程在中间节点失败并保留 checkpoint
- **则** 使用相同 `thread_id` 可从可恢复点继续执行而非全量重跑

#### 场景：边界情况-thread_id 不一致
- **当** 恢复请求提供的 `thread_id` 与历史执行不一致
- **则** 系统拒绝恢复并返回上下文不匹配错误

### 需求：工作流接口必须提供统一错误模型
系统 应当 暴露统一错误对象，至少包含 `error_code`、`error_message`、`failed_node`、`retryable`、`trace_id` 字段，以便上层稳定处理。

#### 场景：节点失败返回统一错误
- **当** 任一节点发生业务或运行时异常
- **则** 返回统一错误模型并附带失败节点标识

#### 场景：边界情况-不可重试错误
- **当** 错误被标记为不可重试
- **则** 系统不得自动重试，并将流程状态设置为终止失败

参考依据：LangGraph Graph API、Persistence、Interrupts 文档。
