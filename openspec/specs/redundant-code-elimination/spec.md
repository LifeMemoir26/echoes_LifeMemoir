## 目的
来自 `change-to-langgraph-core` 的同步规范：清理冗余代码并保持行为等价。

## 需求

### 需求：重复模块必须识别并收敛为单一权威实现
系统 应当 建立重复模块清单并为每类能力保留唯一权威模块，重复命名或同义实现必须下线或重定向。

#### 场景：重复模块识别
- **当** 扫描代码库发现同义实现文件（如命名变体）
- **则** 系统输出清单并标记权威模块与待移除模块

#### 场景：边界情况-重复模块行为不一致
- **当** 两个重复模块行为存在差异
- **则** 系统在移除前要求补充行为对齐策略与回归用例

### 需求：旧编排路径必须在迁移完成后移除
系统 应当 在对应 workflow 完成迁移并通过验收后，移除旧的手工编排入口，避免双系统长期并存。

#### 场景：单流程迁移完成后下线旧入口
- **当** 某主流程通过行为一致性验证
- **则** 该流程旧编排入口被移除或禁用

#### 场景：边界情况-验收未通过
- **当** 新旧路径结果不一致
- **则** 系统保留旧入口并阻止删除动作

### 需求：命名规范必须统一且可自动检查
系统 应当 采用统一命名规范（目录、模块、类型、接口）并提供自动检查规则，禁止新增不规范命名。

#### 场景：新增文件命名校验
- **当** 提交新增模块文件
- **则** 命名检查通过后方可进入后续流程

#### 场景：边界情况-历史命名兼容
- **当** 历史模块暂未迁移完成
- **则** 系统允许兼容映射但必须附迁移截止计划

### 需求：删除冗余代码必须可追溯
系统 应当 在每次冗余移除时记录“移除原因、替代路径、影响范围、回滚方式”。

#### 场景：冗余代码移除记录完整
- **当** 删除任一重复模块
- **则** 变更记录包含原因、替代位置和回滚方案

#### 场景：边界情况-无替代实现
- **当** 模块被判定冗余但无可用替代
- **则** 禁止删除并标记为待设计决策

### 需求：清理后必须保持行为等价
系统 应当 对清理影响范围执行行为回归，保证对外可观察行为不变。

#### 场景：清理后回归通过
- **当** 清理任务完成
- **则** 关键回归用例全部通过且输出等价

#### 场景：边界情况-出现行为回退
- **当** 回归发现输出差异或错误率上升
- **则** 立即回滚对应清理变更并恢复旧实现

参考依据：大型重构最小可回滚实践与变更可追溯原则。
