## 目的
来自 `change-to-langgraph-core` 的同步规范：统一分层架构定义。

## 需求

### 需求：系统必须采用 DDD 四层结构
系统 应当 采用并保持以下目录分层：`src/domain`、`src/application`、`src/infra`、`src/interfaces`，且各层职责唯一。

#### 场景：新模块落位检查
- **当** 新增业务模块
- **则** 模块文件落在对应分层目录且职责与分层定义一致

#### 场景：边界情况-职责冲突模块
- **当** 同一模块同时包含编排逻辑和基础设施调用细节
- **则** 该模块被判定为违规并需拆分后方可通过验收

### 需求：依赖方向必须单向且可审计
系统 应当 满足依赖方向：`interfaces -> application -> domain`，`infra -> domain(接口/模型)`；`domain` 不得依赖外层。

#### 场景：领域层纯净性检查
- **当** 执行依赖扫描
- **则** `domain` 不出现 FastAPI/LangGraph/数据库 SDK 等外层依赖

#### 场景：边界情况-反向依赖
- **当** 发现 `domain` 导入 `infra` 或 `interfaces`
- **则** 构建检查失败并输出违规依赖路径

### 需求：应用层必须承载工作流编排而非基础设施细节
系统 应当 将 LangGraph 节点与流程编排置于 `application` 层，应用层通过抽象接口调用 `infra`，不得直接绑定具体客户端实现细节。

#### 场景：应用层调用仓储抽象
- **当** 工作流节点需要读写存储
- **则** 节点调用仓储接口而非直接调用具体 SQLite/Chroma 客户端

#### 场景：边界情况-直接调用基础设施实现
- **当** 应用层直接依赖具体基础设施类
- **则** 该实现不满足分层要求并需改为接口注入

### 需求：接口层必须保持薄适配
系统 应当 将 `interfaces` 层限定为入参校验、鉴权、DTO 转换和调用应用层，不得放入复杂业务规则。

#### 场景：控制器职责校验
- **当** 审查接口层代码
- **则** 控制器仅包含适配代码，不包含领域决策规则

#### 场景：边界情况-接口层出现业务决策
- **当** 控制器实现包含条件分支决定业务规则
- **则** 该逻辑必须迁移到 `application` 或 `domain`

### 需求：分层接口契约必须显式定义
系统 应当 为跨层调用定义输入/输出 DTO 与错误模型，至少包含 `request_id`、`trace_id`、`error_code` 字段规范。

#### 场景：跨层调用契约一致
- **当** 任一接口层调用应用层用例
- **则** 请求与响应结构符合约定 DTO，错误字段一致

#### 场景：边界情况-缺失追踪字段
- **当** 响应中缺少 `trace_id`
- **则** 契约校验失败并输出缺失字段错误

参考依据：DDD 分层原则与 LangGraph 应用结构建议。
