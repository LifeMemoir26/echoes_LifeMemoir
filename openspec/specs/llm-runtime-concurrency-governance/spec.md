## 目的
来自 `change-to-langgraph-core` 的同步规范：固化 LLM 运行时并发治理能力。

## 需求

### 需求：LLM 运行时必须保留现有并发治理语义
系统 应当 在 `LLMGateway` 中保留 key 轮询、并发上限、429 冷却、重试退避与超时控制语义，迁移后默认行为与现有并发管理器保持一致。

#### 场景：正常并发请求分发
- **当** 多个请求并发进入网关
- **则** 系统按并发上限调度并在可用 key 池内轮询分配

#### 场景：边界情况-仅一个 key 可用
- **当** key 池只有一个可用 key
- **则** 系统仍可串行/受限并发执行且不触发索引越界或死锁

### 需求：429 冷却与重试必须可观测
系统 应当 在检测到 429 时将对应 key 标记冷却并记录解锁时间，重试行为必须记录重试次数、延迟与最终结果。

#### 场景：命中 429 切换 key
- **当** 当前请求命中 429
- **则** 系统将该 key 置为冷却并切换到下一个可用 key 重试

#### 场景：边界情况-全部 key 冷却
- **当** 所有 key 同时处于冷却期
- **则** 系统按最早解锁策略等待或返回可重试错误，且日志包含冷却状态摘要

### 需求：超时与取消必须向下传播
系统 应当 支持节点级超时和流程级超时，取消信号必须向下游调用传播，避免僵尸任务和无限等待。

#### 场景：节点超时取消
- **当** 节点执行超过节点级超时阈值
- **则** 系统取消该请求并返回超时错误，且流程进入受控失败状态

#### 场景：边界情况-上游取消
- **当** 调用方主动取消流程
- **则** 下游未完成任务被取消并释放并发配额

### 需求：运行时接口必须定义明确请求/响应契约
系统 应当 为网关定义统一请求结构（messages/model/temperature/max_tokens/timeout）和响应结构（content/usage/model/latency/error），并保证向后兼容字段。

#### 场景：兼容旧调用字段
- **当** 调用方使用旧版本兼容字段发起请求
- **则** 系统完成字段映射并返回标准化响应

#### 场景：边界情况-非法参数
- **当** 请求参数超出允许范围（如负超时）
- **则** 系统在入口校验阶段直接返回参数错误，不调用外部模型

### 需求：并发观测指标必须统一输出
系统 应当 输出最少指标集合：`total_requests`、`successful_requests`、`failed_requests`、`average_latency_ms`、`retry_count`、`cooldown_events`。

#### 场景：周期性指标采集
- **当** 执行窗口结束或触发采集周期
- **则** 系统可读取完整指标快照且字段齐全

#### 场景：边界情况-无请求窗口
- **当** 统计窗口内无任何请求
- **则** 指标输出为零值且不报错

参考依据：LangGraph Runtime 与生产化执行实践；现有并发管理行为基线。
